# Special-Purpose Pages

Themes can include their own pre-defined pages via `theme.pagesDir`. These pages follow standard routing rules; however, user-provided pages always take priority in the event of a route conflict.

This feature is ideal for implementing functional pages such as:

- Custom landing pages
- Dedicated index or menu screens
- Link directories or "Friends" pages
- Contact forms
- Theme-specific `/404` and `/offline` fallbacks

## Providing Pages from a Theme

Configure the directory in `theme/index.js`:

```js
export default () => ({
	theme: {
		root: '.',
		pagesDir: './pages'
	}
})
```

Typical `theme/pages/` structure:

```
pages/
  index.mdx
  menu.mdx
  friends.mdx
  contact.mdx
  404.mdx
  offline.mdx
```

## Route-Based Branching

If you prefer fixed rules, you can implement logic based on the route path:

```js
const route = ctx.page.routePath
if (route === '/menu') { /* ... */ }
```

*Note: Directory index routes include a trailing slash in `routePath` (e.g., `/guide/`).*

## Selecting Layouts via Frontmatter

The most intuitive method is to define a `layout` property in your frontmatter and render accordingly within your template.

Example `menu.mdx`:

```mdx
---
title: Menu
layout: menu
hidden: true
---

# Menu

This page utilizes a differentiated layout.
```

Template implementation:

```jsx
const MenuLayout = ({ ctx, PageContent }) => (
	<div class="menu-layout">
		<h1>{ctx.page.title}</h1>
		<PageContent />
	</div>
)

const DocLayout = ({ ctx, PageContent }) => (
	<div class="doc-layout">
		<aside>{/* navigation */}</aside>
		<main>
			<PageContent />
		</main>
	</div>
)

export default function PageTemplate({ PageContent, ExtraHead, ctx }) {
	const layout = ctx.page.frontmatter?.layout
	return (
		<>
			<html>
				<head>
					<ExtraHead />
					<title>{ctx.page.title || ctx.site.name}</title>
				</head>
				<body>
					{layout === 'menu' ? (
						<MenuLayout ctx={ctx} PageContent={PageContent} />
					) : (
						<DocLayout ctx={ctx} PageContent={PageContent} />
					)}
				</body>
			</html>
		</>
	)
}
```

## Interactive Pages (Forms, etc.)

For pages requiring complex interactivity, such as contact forms, it is recommended to render the page content normally while providing a client-only component (e.g., `ContactForm.client.jsx`) in the theme's `components/` directory for use within the page body.

## Alternative Strategy: Functional HTML Pages

Special-purpose pages do not require MDX. You can place pure `.html` files in `pages/` (or the theme's `pagesDir`) to implement standalone landing pages, embedded forms, or lightweight frontend applications.

These files are processed by Vite and passed through `theme.template`. You can then utilize the template to stream these into a "standalone" layoutâ€”for instance, by omitting sidebars, tables of contents, or applying custom `<head>` tags.

Common branching patterns:

- **By file extension**: `ctx.page.path.endsWith('.html')`
- **By route**: `ctx.page.routePath === '/menu'`
- **By frontmatter**: `ctx.page.frontmatter?.layout === 'landing'`

## Asset and Base Path Considerations

Typically, you do not need to apply `withBase()` to `theme.sources` URLs or static files in `public/`, as Methanol resolves these during build. However, `withBase('/...')` remains a reliable safeguard when generating URLs dynamically in JavaScript where development-time resolution might differ.
